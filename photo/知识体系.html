    • HTTP协议
      • 版本区别
        • HTTP 1.0
          • 优点
            • 可以发送任何形式的内容: 文字、图片、视频和二进制,丰富了Post,Get和Head请求和响应格式,加入头信息
          • 缺点
            • 每个TCP只能发送一次请求,TCP连接非常耗时
        • HTTP 1.1
          • 优点
            • 引入持久连接,TCP连接默认不关闭,对于同一个域名,默认支持6个持久连接
            • 引入管道机制,即一个持久连接中,可以发送多个请求,但是响应还是有顺序
          • 缺点
            • TCP响应有顺序,必须等待前一个请求的返回,服务器才能处理下一个请求,如果某一个请求特别慢,后面的请求就会排队等待(队头阻塞)
        • HTTP 2.0
          • 特点
            • 二进制,多工,服务器推送,头部信息压缩
    • HTTP请求和响应格式
      • 请求
        • Host:指定服务器域名,可以访问服务器上的不同服务
        • connection keep-alive不关闭TCP连接,close 关闭TCP连接
        • accept-Encoding: 表示可以接受的压缩方式
        • user-agent: 区分客户端的操作系统
      • 响应
        • content-type: 服务端告诉客户端数据类型,文本,图片,视频,文件等
        • content-Encoding:数据采用的压缩格式
        • Transform-Encoding:chunks表示分块传递,需要Content-Length
        • Content0Lenght:声明数据长度
    • 建立连接和断开连接
      • 三次握手
        • 客户端发送SYN seq=x
        • 服务端向客户端发送SYN seq=y ACK=x+1
        • 客户端向服务端发送ACK =y+1
      • 四次挥手
        • 客户端向服务端发送FIN seq= x+2
        • 服务端向客户端发送ACK seq=x+3
        • 服务端向客户端发送FIN seq=y+1
        • 客户端向服务端发送ACK seq= y+2
    • HTTPS/SSL(Secure Socket Layer)
      • 概念
        • Https是以安全为目标的通信协议,在http的下一层添加了SSL层
      • 原理
        • 采用了非对称加密算法（RSA),为了防止公钥被篡改,把公钥放在数字证书中,证书可信则公钥可信.非对称加密的计算量非常大,为了提高效率,服务端和客户端协商公钥的过程使用非对称加密,对话过程使用公钥加密
      • 过程
        • 客户端发送协议的版本号,一个随机数A和支持的加密方式
        • 服务端确认加密的方式,发送数字证书和随机数B
        • 客户端确认数字证书有效,使用公钥对一个随机数C进行加密,然后发送给服务器
        • 服务器使用私钥对C进行解密
        • 客户端和服务端使用约定好的加密方式,对随机数ABC生成密钥,以后的通话就使用这个密钥
    • HTTP与HTTPS区别
      • HTTPS需要CA申请证书,需要一定费用
      • HTTP采用超文本协议,一般使用明文传递信息,HTTPS具有安全协议SSL加密传输协议
      • 采用的端口不一样,HTTP采用80端口,HTTPS采用443端口
      • HTTP是无状态的,HTTP+SSL对信息进行加密,身份验证更加安全
      • HTTP建立连接的速度比HTTPS要快,因为HTTPS除了三次握手还需要SSL握手

    • 数据集合
      • ArrayList 对象数组
      • LinkedList 对象链表 内部类Node,指向前后
      • HashMap
        • 原理: 是一个键值对存储的数据结构,其内部结构是一个散列表,当执行Put操作的时候,会根据Key计算出的hash值,在根据hash值计算出桶的位置,如果这个位置上为null，则将value放置进去,如果该节点有值,则遍历单链表或者红黑树,如果hash值和key值相同,则替换原来的值,如果没有,则放到单链表或者红黑树的尾部,最后如果超过阈值,则扩容两倍,当执行get时候,会根据key值计算hash值,再计算桶的位置,如果该位置为null，则返回Null，如果有值,则遍历单链表或者红黑树,返回响应
        • key的高16位与低16位进行异或操作,可以减少碰撞的发生
        • 容器的大小为2的幂次方,可以更加均匀的计算桶的位置,减少碰撞的发生
      • LinkedHashMap
      • HashSet: 内置HashMap,使用key来实现值的唯一性
      • ConcurrentHashMap
        • Java 1.7
          • 使用锁分段技术,初始16个segment， 采用ReentrantLock,乐观锁,尝试获取锁失败,会不断尝试,当达到阈值时,会阻塞等待。真正存放数据的地方是HashEntry 类似HashMap
        • Java 1.8
          • 采用CAS和Sychronized 来保证线程同步,get的时候不加锁,使用validate来保证变量的原子性,其他和hashmap类似
      • SpraseArrayList,ArrayMap和HashMap的对比
        • 优点
          • 对于基础类型不需要装箱
          • 采用了数组,内存占用小
        • 缺点
          • 查找key是二分查找,相比较效率比较低,使用空间换效率
    • 线程与并发
      • 线程
        • Thread 通过start开启线程,Sleep不会释放锁,wait和join会释放锁,sleep和join可以被中断
        • Runnable Callable 是线程运行的载体,区别是Runnable没有返回值,Callable有返回值
      • 锁
        • ReentrantLock 是公平锁也是可重入锁,是代码实现的锁机制，线程池的等待队列就是使用ReenterantLock实现的
        • Sychronized,是JVM实现的锁,是非公平锁,可以对方法或者方法块进行加锁,可以使用Thread的sleep和wait来阻塞线程,sleep是不释放锁,wait是释放锁
        • Volatile作用在变量上,原理是,会在字节码上一个指令，是内存屏蔽,禁止指令重排序,读取变量的时候强制读取主内存的数据,写数据的时候,会将其他线程的缓存置为失效.不支持非原子性的操作,例如int i = a++
      • 线程池
        • 线程池的好处
          • 重用,减少创建和销毁所产生的性能开销
          • 避免过多的线程争抢CPU而产生的阻塞现象
          • 可以做统一的管理
        • ThreadPoolExecutor的几个参数
          • corePoolSize 核心线程数,不会被销毁
          • maximumPoolSize 最大线程数,在达到最大线程数后,会进入阻塞队列
          • keepAliveTime 非核心线程数, 在空转时,存活的时间
          • Unit 时间单位
          • workQueue 阻塞队列
          • threadFactory 创建线程的工厂
          • rejectHandler 拒绝策略,当任务队列满了情况,会调用这个Handler 一般会抛出一个rejectedExecutionExecption异常
        • 常用的几个线程池
          • SingleThreadPool 仅仅是有一个核心线程,用来执行串行任务
          • FixedThreadPool 只有固定数量的核心线程
          • CachedThreadPool 只有非核心线程
          • ScheduleThreadPool 有核心和非核心线程数

  • 性能优化
    • 常见的性能问题
      • 内存泄露
        • Android内存分配方式
      • OOM(内存溢出)
        • 大Bitmap、列表Bitmap
      • 耗电量问题
        • 定位、动画
      • 内存抖动
        • 频繁GC
    • 造成性能问题常见的原因
      • 在ui线程中进行了耗时的操作,导致ui线程卡顿
      • ui布局过于复杂,无法再16ms内完成渲染
      • view过度绘制,导致不必要的区域,多次渲染,造成GPU或者CPU负载过重
      • 频繁的GC，导致ui线程卡顿
      • 设置子线程的优先级高于UI线程的优先级,造成抢占ui线程CPU时间片,阻塞渲染操作
      • 同一时间 执行的动画太多、未能正确使用硬件加速,绘制了不要的区域,造成GPU或者CPU负载过重
      • 冗余的资源和逻辑,导致加载和执行率缓慢
      • 单例、静态变量持有引用、非静态内部类、Handler等
    • 如何解决这些问题
      • 检查UI线程中是否做了耗时操作,开启Strictmode模式
        • 如果主线程中有网络或者IO操作，Logcat中会有tag为.d/strictmode的日志
      • 使用ConstraintLayout来解决复杂的布局问题
        • 1.0之前的版本性能较差,在简单布局中绘制的效率不如LinearLayout和FrameLayout,1.1之后优化了相关的性能操作,添加了一些新的动能
      • 在开发者模式中,开启GPU过度绘制,对过度绘制区域进行优化,canvas.clipRect函数和多余的背景设置
      • 使用BlockCanary来检测UI卡顿的问题
        • Choreographer,OnFrameMetricsAvailableListener
      • 使用LeakCanary来检测内存泄露的问题
      • ViewStub和Merge的使用

        • sharedPreferences
          • 简介
            • Android中常用的存储方式,可以用用来存储小的键值对,以xml的方式保存文件

            • 跨进程不安全,由于没有使用跨进程的锁,就算使用MODE_MULTI_PROCESS，也不能保证跨进程安全，跨进程读取频繁,可能造成文件全部丢失

    • Dalvik和ART
      • 发展史
        • Android 2.2
          • Android首次在 Dalvik上使用JIT（Just in Time Compiler) 在运行时对dex/odex进行编译与优化,将Dalvik Code编译为精简的Native Code，JIT的引入,将Dalvik的运行性能提升了3-6倍
          • 缺点：每次运行都需要进行编译,因为每次都需要编译,所以会造成额外的电池消耗
        • Android 4.4
          • Android带来了全新的运行环境ART（Android run time)和全新的编译策略AOT（Ahead of time)
          • AOT是静态编译,在安装时,会启动dex2oat过程,将dex文件预编译为ELF文件,每次运行时不需要重新编译
        • Android 5.0
          • 全面替换ART运行环境,编译策略也改为AOT
          • 缺点:首次安装时间会变长,会额外增加文件大小,一个100M的APP，预编译可能达到200M
        • Android 7.0
          • 运行环境任然是ART,但是编译策略改为AOT和JIT混合编译
          • 特点：在首次安装时,不会进行预编译,直接运行,使用运行时编译,在手机 处于空闲或者充电状态时,在进行预编译
          • 优点:集合了AOT和JIT的优点,使得安装速度加快,运行速度,存储空间和耗电量等指标都得到了优化
      • ART相遇对Dalvik的提升
        • ART在标记所有对象的时候不需要挂起所有的线程,ART中的GC要求对象在创建时候,标记自身的堆栈,由于时间非常的短,不需要挂起线程
        • 提供了LOS（Large Object Space) 专门为bitmap的分配提供了空间,提高了内存的管理效率和整体效率
        • ART中有moving collector来压缩对象,使得内存变得更加紧凑
